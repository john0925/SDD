## Context

當前系統已具備餐點展示與基礎 API，但購物車按鈕無功能，且缺乏頁面間的導航與數據傳遞機制。我們需要建立一套完整的流程，並將複雜的營養評分邏輯封裝在前端狀態與後端回傳值之間。

## Goals / Non-Goals

**Goals:**
- 實作 Vue Router 導航架構（Home, Checkout, Success）。
- 使用 Pinia 實作購物車狀態，支援營養數據的即時聚合。
- 實作基於歐幾里得距離的智慧評分演算法。
- 模擬訂單建立 API 並回傳評分結果。

**Non-Goals:**
- 不涉及使用者身分驗證（假設為單一本地使用者）。
- 不涉及真實金流支付系統。
- 不涉及後端資料庫存取（維持 JSON 模擬）。

## Decisions

### 1. 狀態管理選擇：Pinia
- **決定**：使用 Pinia 作為全域狀態中心。
- **理由**：相較於 Vuex，Pinia 的 API 更簡潔且對 TypeScript 支援更完美。它能輕鬆處理購物車的增刪改查，並透過 Getters 即時加總營養素。
- **替代方案**：Composition API (reactive) 雖然也能做到，但在跨頁面調度與除錯工具（DevTools）的支援上不如 Pinia 強大。

### 2. 路由導航：Vue Router
- **決定**：實作單頁應用程式（SPA）路由。
- **理由**：為了還原 `checkout.html` 與 `success.html` 的無縫轉場體驗，避免全頁面重新整理造成的狀態遺失。

### 3. 營養評分位置：前端計算 vs 後端回傳
- **決定**：前端即時預覽，後端確認回傳。
- **理由**：前端負責結帳頁面的「預計得分」即時反饋，增進互動感；後端在 `POST /api/orders` 時執行最終計算並給予訂單編號，確保資料的權威性。

## Risks / Trade-offs

- **[風險] 浮點數精度問題** ──▶ 使用歐幾里得距離涉及開根號，計算佔比時需四捨五入至小數點後一位以維持展示一致性。
- **[權衡] 靜態 JSON 限制** ──▶ 由於無資料庫，訂單編號將採用隨機產生的方式，且重新整理頁面後訂單紀錄將會消失。
