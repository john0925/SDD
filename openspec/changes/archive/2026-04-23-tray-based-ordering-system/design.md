## Context

目前的購物車（`cart.ts`）採用扁平的 `items` 陣列。為了支援多人點餐與精準營養分析，我們需要將資料結構升級為巢狀的「托盤」模式。每個托盤獨立計算營養與分數，而訂單總計則彙整所有托盤的金額。

## Goals / Non-Goals

**Goals:**
- 重構 Pinia Store，導入 `trays` 結構。
- 實作下拉式托盤選擇器（Tray Selector）。
- 重設計結帳頁面佈局，實現托盤分組展示與獨立評分。
- 維持 `DESIGN.md` 的「無線化」與「分層深度」原則。

**Non-Goals:**
- 不增加繁瑣的用戶設定（如飲食目標設定）。
- 不變動後端 JSON 資料庫的基本欄位（僅調整請求資料結構）。

## Decisions

### 1. Pinia Store 結構重構
- **決定**：`state` 包含 `trays: Tray[]` 與 `activeTrayId: string`。
- **理由**：巢狀結構能自然地將餐點與特定用餐者（托盤）綁定，方便計算個別得分。
- **初始化**：系統啟動時若無托盤，自動執行 `addTray()`。

### 2. 托盤選擇器組件 (TraySelector)
- **決定**：使用帶有 `backdrop-blur` 效果的下拉選單。
- **理由**：避免橫向捲動造成的導航負擔，特別是在行動裝置上。下拉選單能更清晰地標示當前「作用中」的托盤。

### 3. 托盤式結帳佈局
- **決定**：每個托盤為一個大卡片（`surface-container-low`），內部餐點與營養摘要使用 `surface-container-lowest` 背景區隔。
- **佈局**：採用不對稱設計，左側列出餐點，右側（或下方）展示該托盤的評分與數據。

### 4. 評分引擎升級
- **決定**：將原有的 `nutritionScore` 改為接收 `Tray` 物件作為參數的純函式，讓多個托盤能同時獨立計算。

## Risks / Trade-offs

- **[風險] 托盤切換的遺忘** ──▶ 使用者可能忘記切換托盤而將餐點加錯位置。提供明顯的當前托盤標示與「轉移托盤」功能作為緩解。
- **[權衡] UI 複雜度** ──▶ 托盤化會增加結帳頁面的垂直長度，需確保捲動體驗流暢且標題清晰。
